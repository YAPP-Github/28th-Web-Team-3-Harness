# 공통 AI 에이전트 Harness 설계

## 목적

FE와 BE 에이전트가 같은 기획적 업무 흐름, 도메인 정책, 용어, 특이사항을 기준으로 작업하게 한다. Harness는 기술 구현 규칙이나 기획 원본을 복제하지 않고, 확정된 내용을 개발 및 에이전트용 컨텍스트로 정규화한다.

## 저장소 구조

```text
.
├── AGENTS.md
├── CLAUDE.md
├── README.md
├── .agents/
│   ├── agents/
│   └── skills/
├── .claude -> .agents
├── domain/
│   ├── glossary.md
│   ├── policies/
│   ├── workflows/
│   └── exceptions.md
├── decisions/
└── templates/
```

`AGENTS.md`는 모든 에이전트의 공통 진입점이다. `CLAUDE.md`는 Claude Code가 `AGENTS.md`를 읽도록 연결하는 얇은 어댑터다. `.agents/`가 에이전트와 스킬 정의의 단일 원본이며, `.claude`는 `.agents`를 가리키는 심볼릭 링크다.

## 콘텐츠 경계

Harness에 포함한다.

- 공통 도메인 용어
- 확정된 비즈니스 정책과 불변조건
- 유저 및 업무 플로우
- 예외와 알려진 특이사항
- 공통 에이전트 역할
- FE와 BE가 함께 사용하는 스킬
- 정책 변경 결정과 이유
- 정책 및 플로우 문서 템플릿

Harness에 포함하지 않는다.

- FE 또는 BE에만 적용되는 구현 규칙
- 실제 시크릿과 인증 정보
- 확정되지 않은 기획 아이디어
- 자주 변하는 API 명세의 복제본
- 완료된 임시 작업 메모

## 참조와 버전 관리

FE와 BE는 Harness를 Git 서브모듈로 사용한다. 각 저장소는 검증된 Harness 커밋을 고정하며, 원격 `main`을 자동 추적하지 않는다. Harness 변경이 병합되면 FE와 BE에서 서브모듈 포인터를 갱신하는 별도 PR을 만든다.

정책 문서는 안정적인 식별자를 사용한다. 예: `POLICY-AUTH-001`. 워크플로 문서는 관련 정책 식별자를 링크한다. FE와 BE 작업은 관련 정책 식별자를 PR 설명에 기록할 수 있다.

## 변경 검토

Git 변경은 FE 리뷰어 1명과 BE 리뷰어 1명이 검토한다. 기획 프로세스와 기획 원본 관리는 Harness 범위 밖이다. 기획 확인이 필요하면 팀이 사용하는 외부 절차를 따르고, 개발자가 확정된 결과를 Harness에 반영한다.

## 초기 검증 전략

초기에는 별도 `scripts/` 디렉터리나 자동 검증기를 두지 않는다. 문서 수와 변경 빈도가 낮은 동안 PR 리뷰와 템플릿으로 품질을 관리한다. 깨진 링크, 필수 항목 누락, 정책 ID 충돌이 반복되면 자동 검증을 추가한다.

## 초기 완료 조건

- 진입점 문서 `AGENTS.md`, `CLAUDE.md`, `README.md`가 존재한다.
- `.agents/`가 에이전트와 스킬의 단일 원본이다.
- `.claude`가 `.agents`를 가리키는 상대 심볼릭 링크다.
- 도메인, 결정, 템플릿 디렉터리가 책임별로 분리된다.
- 각 디렉터리의 목적과 작성 규칙이 진입점 문서에서 설명된다.
